# Architectural Decisions

## Devcontainer Feature Artifact Strategy

### Decision
Use tar.gz artifact containing dotfiles for devcontainer feature. The tar.gz is extracted to a temporary directory for running the install script, then cleaned up. The ~/ubuntu-dotfiles repository is cloned fresh from the remote at the latest version.

### Context
The devcontainer feature needs to:
1. Be testable locally without requiring a push to the remote
2. Ensure consistency across all installations
3. Provide the install script and dotfiles to the container

### Problem with commit hash approach
- Requires pushing commits before testing locally
- Creates a dependency on remote availability during development
- Makes the test-before-commit workflow difficult
- Couples the feature version to a specific commit hash

### Solution
- Bundle dotfiles as a tar.gz artifact (generated by `make sync`)
- Extract tar.gz to a temporary directory for the feature install
- Clean up the temporary directory after installation
- Clone ~/ubuntu-dotfiles from remote at the latest version for the actual working repository

### Benefits
- Can test locally without pushing
- Consistent behavior: same tar.gz content across all runs
- Clean separation: feature artifacts vs. working repository
- Install script still gets latest updates via git clone
- Temporary directory is isolated and cleaned up automatically
